## 🐛 Bugs Identified

### Bug #1: DateTimeParseException — YearMonth Parsing Error

**Severity:** 🔴 **CRITICAL**  
**Impact:** HR cannot view candidate details (500 error)  

**Error Log:**
```
java.time.format.DateTimeParseException: Text '2019-01-01' could not be parsed, unparsed text found at index 7
        at java.time.YearMonth.parse(YearMonth.java:282)
        at com.hcmute.careergraph.controllers.CandidateController.calcTotalYearsSimple(CandidateController.java:294)
```

**Root Cause:**
- `CandidateController.calcTotalYearsSimple()` sử dụng `YearMonth.parse(exp.startDate())`
- Database lưu experience dates theo format `"2019-01-01"` (YYYY-MM-DD, full date)
- `YearMonth.parse()` chỉ chấp nhận format `"2019-01"` (YYYY-MM)
- Frontend gửi full date từ date picker → backend parsing fail

**Affected Workflow:**
```
User creates candidate profile
  → Adds experience: "2019-01-01" to "2021-12-31"
  → Saves to database
  → HR views candidate detail
  → Backend tries to parse "2019-01-01" as YearMonth
  → Exception thrown → 500 error
```

**Test Case to Reproduce:**
1. Create candidate with ID: `837ebdc4-721a-4133-b511-481b03d951fd` (Thịnh Lương)
2. Upload CV: Java_Developer_Fresher_CV_Template.pdf
3. Add experience với date format YYYY-MM-DD
4. HR tìm kiếm candidate
5. Click vào candidate detail → 500 error

## ✅ Fixes Implemented

### Fix #1: Add Flexible Date Parser

**File:** `CandidateController.java`

**Changes:**
1. Added helper method `parseToYearMonth()` to handle both formats:
   ```java
   /**
    * Parse date string to YearMonth, handling both formats:
    * - "2019-01" (YYYY-MM)
    * - "2019-01-01" (YYYY-MM-DD)
    */
   private YearMonth parseToYearMonth(String dateStr) {
       if (dateStr == null || dateStr.isBlank()) {
           return YearMonth.now();
       }
       
       try {
           // Try YYYY-MM format first
           if (dateStr.length() == 7) {
               return YearMonth.parse(dateStr);
           }
           // Handle YYYY-MM-DD format (convert LocalDate to YearMonth)
           if (dateStr.length() == 10) {
               return YearMonth.from(LocalDate.parse(dateStr));
           }
           // Fallback: parse as LocalDate then convert
           return YearMonth.from(LocalDate.parse(dateStr));
       } catch (Exception e) {
           log.warn("Failed to parse date '{}', using current month: {}", 
                    dateStr, e.getMessage());
           return YearMonth.now();
       }
   }
   ```

2. Updated `calcTotalYearsSimple()` to use new parser:
   ```java
   YearMonth start = parseToYearMonth(exp.startDate());
   YearMonth end = Boolean.TRUE.equals(exp.isCurrent())
           ? YearMonth.now()
           : parseToYearMonth(exp.endDate());
   ```

3. Added logging for debugging

4. Added `@Slf4j` annotation to enable logging

**Benefits:**
- ✅ Backward compatible with both date formats
- ✅ Graceful fallback to current month if parsing fails
- ✅ Logging for debugging future issues
- ✅ No breaking changes to frontend

### Fix #2: Remove Local Pagination Slice

**File:** `CandidateHorizontalList.tsx`

**Changes:**
```tsx
// BEFORE (lines 25-36)
const PAGE_SIZE = 3;
const visibleCandidates = useMemo(() => {
    return isMobile ? candidates : candidates.slice(0, PAGE_SIZE);
}, [candidates, isMobile]);

// AFTER
const visibleCandidates = candidates; // Backend đã handle pagination
```

**Rationale:**
- Backend returns exactly `size` candidates per page (default 10)
- Frontend should display ALL candidates from backend response
- Pagination handled by backend via `page` and `size` parameters
- Frontend only needs to trigger `onPageChange(newPage)` for next/prev

**Benefits:**
- ✅ All 10 candidates displayed correctly
- ✅ Pagination accurate (Page 1/2 when totalElements > 10)
- ✅ Simpler code, less confusion

```tsx
    </div>
)}
```

**Benefits:**
- ✅ Clear visual feedback during search
- ✅ Preserves old results while fetching (no jarring blank screen)
- ✅ Different states for initial load vs pagination
- ✅ Better UX with color-coded loading indicators

## 🔍 Root Cause Analysis

### Why These Bugs Occurred

**Bug #1: Date Parsing**
- **Cause:** Frontend and backend date format mismatch
- **Why not caught:** No validation on date format at API boundary
- **Prevention:** Add DTO validation for date formats, or standardize to ISO-8601

**Bug #2: Pagination**
- **Cause:** Misunderstanding between "pagination at backend" vs "pagination at frontend"
- **Why not caught:** Comment said "don't slice" but code still sliced
- **Prevention:** Code review, automated tests checking rendered count

**Bug #3: Loading State**
- **Cause:** Single `loading` flag for multiple states (initial vs update)
- **Why not caught:** Dev testing didn't notice UX issue
- **Prevention:** UX review, usability testing

---

## 💡 Lessons Learned

1. **Date Formats Are Tricky:**
   - Always handle multiple formats gracefully
   - Log warnings instead of throwing exceptions
   - Add fallback logic for unparseable data

2. **Backend Pagination != Frontend Pagination:**
   - When backend handles pagination, frontend should NOT slice again
   - Clear separation of concerns: backend = data, frontend = display

3. **Loading States Need Nuance:**
   - Different UX for "initial load" vs "fetching more"
   - Preserve old data while fetching new (optimistic UI)
   - Visual feedback at every step

4. **Comments Alone Are Not Enough:**
   - Comment said "don't slice" but code still sliced
   - **Fix:** Remove the problematic code, not just comment it
